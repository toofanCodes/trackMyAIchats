I can certainly give you a more complicated SQL function example! Let's create a function that calculates the average salary for a given department, but only for employees who have worked for the company for a certain number of years, and optionally includes a bonus calculation.

Complicated Example: DepartmentAverageSalaryWithExperienceAndBonus Function

SQL
CREATE OR REPLACE FUNCTION DepartmentAverageSalaryWithExperienceAndBonus (
        p_department_name VARCHAR,
        p_min_years_experience INT,
        p_include_bonus BOOLEAN DEFAULT FALSE
    )
    RETURNS TABLE (
        department_name VARCHAR,
        average_salary NUMERIC,
        total_eligible_employees BIGINT
    ) AS
    $$
    DECLARE
        v_bonus_percentage NUMERIC := 0.10; -- 10% bonus
        v_total_salary_sum NUMERIC := 0;
        v_employee_count BIGINT := 0;
        v_avg_salary NUMERIC;
    BEGIN
        -- Input validation
        IF p_min_years_experience < 0 THEN
            RAISE EXCEPTION 'Minimum years of experience cannot be negative.';
        END IF;
    
        -- Calculate the sum of salaries and count of eligible employees
        SELECT
            SUM(e.salary + CASE WHEN p_include_bonus THEN e.salary * v_bonus_percentage ELSE 0 END),
            COUNT(e.employee_id)
        INTO
            v_total_salary_sum,
            v_employee_count
        FROM
            "Employee" AS e
        JOIN
            "Department" AS d ON e.department_id = d.department_id
        WHERE
            d.department_name = p_department_name AND
            EXTRACT(YEAR FROM AGE(CURRENT_DATE, e.hire_date)) >= p_min_years_experience;
    
        -- Calculate average salary
        IF v_employee_count > 0 THEN
            v_avg_salary := v_total_salary_sum / v_employee_count;
        ELSE
            v_avg_salary := 0; -- Or NULL, depending on desired behavior for no employees
        END IF;
    
        -- Return the results as a table
        RETURN QUERY SELECT
            p_department_name AS department_name,
            v_avg_salary AS average_salary,
            v_employee_count AS total_eligible_employees;
    
    END;
    $$ LANGUAGE plpgsql;

RETURNS TABLE, can something else be returned?

Yes, besides RETURNS TABLE, a PostgreSQL function can return several other types of values:

Other Return Types for PostgreSQL Functions

1. Scalar Types

Functions can return a single value of any standard SQL data type. This is the most common and simplest return type.

  • RETURNS datatype: Returns a single value of a specified data type (e.g., INT, TEXT, DATE, BOOLEAN, NUMERIC).

    Example:

    SQL
    CREATE FUNCTION GetEmployeeCount() RETURNS BIGINT AS $$
    BEGIN
        RETURN (SELECT COUNT(*) FROM "Employee");
    END;
    $$ LANGUAGE plpgsql;

    This function would return a single BIGINT representing the total number of employees.

2. Composite Types (ROW Type)

Functions can return a single row, where the row's structure is defined by a named composite type (which you might have created as a CREATE TYPE or implicitly from a table's row structure) or by explicitly listing the column names and types.

  • RETURNS record: Returns a single row where the column names and types are not predefined by the function but are determined by the query inside the function. The caller must provide an explicit column definition list in the FROM clause when calling such a function.

    Example:

    SQL
    CREATE FUNCTION GetEmployeeDetails(employee_id INT) RETURNS record AS $$
    BEGIN
        RETURN (SELECT employee_name, salary FROM "Employee" WHERE employee_id = $1);
    END;
    $$ LANGUAGE plpgsql;
    
    -- How to call it:
    SELECT * FROM GetEmployeeDetails(101) AS (name TEXT, pay NUMERIC);

  • RETURNS SETOF datatype: Returns a set (multiple rows) of a single scalar data type.

    Example:

    SQL
    CREATE FUNCTION GetAllDepartmentNames() RETURNS SETOF VARCHAR AS $$
    BEGIN
        RETURN QUERY SELECT DISTINCT department_name FROM "Department";
    END;
    $$ LANGUAGE plpgsql;

    This would return a list of department names, one per row.

  • RETURNS SETOF record: Returns a set (multiple rows) where each row is a composite type, but the structure is not predefined by the function. Similar to RETURNS record, the caller defines the column structure.

    Example:

    SQL
    CREATE FUNCTION GetTopPaidEmployees(limit_count INT) RETURNS SETOF record AS $$
    BEGIN
        RETURN QUERY SELECT employee_name, salary FROM "Employee" ORDER BY salary DESC LIMIT limit_count;
    END;
    $$ LANGUAGE plpgsql;
    
    -- How to call it:
    SELECT * FROM GetTopPaidEmployees(5) AS (name TEXT, pay NUMERIC);

3. VOID

A function can return nothing at all. This is useful for functions that perform actions like data modification (inserts, updates, deletes) but don't need to return any data.

  • RETURNS VOID: Indicates the function does not return any value.

    Example:

    SQL
    CREATE FUNCTION LogActivity(activity_desc TEXT) RETURNS VOID AS $$
    BEGIN
        INSERT INTO "ActivityLog" (description, log_time) VALUES (activity_desc, NOW());
    END;
    $$ LANGUAGE plpgsql;

    This function simply inserts a record into a log table and doesn't return any data.

The choice of return type depends entirely on what you want the function to produce: a single value, a single row, a set of single values, or a table of multiple rows and columns, or no value at all.

You're right to notice that! The Pandas sample dataset I provided doesn't explicitly include an employee_id column in the df_employees DataFrame.

Here's why and what it means:

  1. SQL SERIAL PRIMARY KEY: In the SQL CREATE TABLE "Employee" statement, employee_id SERIAL PRIMARY KEY means that PostgreSQL automatically generates a unique, sequential integer for employee_id for each new row inserted into the table. When you feed data from Pandas into a database, if the employee_id column is defined as SERIAL in your database table, you typically don't include it in your Pandas DataFrame, as the database handles its generation.

  2. Pandas DataFrames for Counting:

    • The COUNT(e.employee_id) in the SQL function is used to count the number of rows (employees) that meet the criteria.

    • In a Pandas DataFrame, if you just need to count the number of rows, you don't necessarily need a specific employee_id column. You could count any non-null column, or simply count the rows of the filtered DataFrame. For example, len(filtered_df) or filtered_df['employee_name'].count() would achieve a similar counting purpose in Pandas.

So, while the employee_id is crucial for the database's primary key and for referencing individual employees, it's often omitted from the Pandas DataFrame if the database is responsible for its auto-generation during insertion. If you needed to have a unique identifier in your Pandas DataFrame for other processing before database insertion, you could add one (e.g., by using df.reset_index() to make the index a column, or by generating a simple sequence).

why use INTO here rather than just using AS

You're asking about the difference between INTO and AS in the context of the SELECT statement in your Canvas document. This is a great question that highlights a key distinction in PL/pgSQL!

In the selected line, and specifically within the SELECT ... INTO ... block, INTO is used because you are performing a procedural assignment of query results to local variables.

Here's the breakdown:

  • INTO (for variable assignment in PL/pgSQL):

    • When you use SELECT ... INTO variable1, variable2 ..., you are telling PostgreSQL to execute the SELECT query and then take the values from the first row of the query's result set and assign them directly to the specified PL/pgSQL variables (totalSal and emplCnt in your case).

    • This is a procedural operation, meaning it's about storing data within the function's execution context for further processing (like calculating the average).

    • It's used when you expect the SELECT query to return at most one row, or you only care about the first row if multiple are returned (though typically you'd design such SELECT INTO queries to return exactly one row).

  • AS (for aliasing in SQL queries):

    • AS is used for aliasing columns or tables within a standard SQL SELECT statement. It gives a temporary name to a column or a table in the output result set of a query.

    • For example, in your RETURN QUERY SELECT depName AS department_name, ..., AS department_name renames the depName column to department_name in the table that the function returns. This is about how the output of the query is presented, not about storing values into variables for internal use.

In summary:

  • You use INTO when you want to capture the results of a SELECT query and store them into local variables within your PL/pgSQL function for subsequent procedural logic.

  • You use AS when you want to give a temporary name (alias) to a column or table in the result set of a SQL query.

Your code correctly uses INTO for assigning the SUM and COUNT to totalSal and emplCnt variables, and AS for defining the output column names when the function returns its final table.

where do these functions persist? are they stored in the database? (compare to stored procedures ehre)

Yes, the DeptAvgSalary function, like all functions you create using CREATE FUNCTION in PostgreSQL, is stored directly in the database.

They are persistent database objects, just like tables, views, indexes, or sequences. Once you create a function, it's saved within the database schema and remains there until you explicitly drop it using DROP FUNCTION. This means anyone with appropriate permissions can call and reuse the function.

Comparison to Stored Procedures:

The term "stored procedure" is more commonly used in other relational database management systems (RDBMS) like SQL Server, Oracle, or MySQL. In those systems:

  • Stored Procedures typically refer to routines that perform a set of SQL statements, often involving data modification (INSERT, UPDATE, DELETE), and may or may not return a value. They are primarily for executing logic.

  • Functions (in those contexts) are usually expected to return a single scalar value and are often used within SELECT statements or expressions.

In PostgreSQL:

  • The term "Function" (created with CREATE FUNCTION) is very broad and encompasses what other databases call both functions and stored procedures. PostgreSQL functions can:

    • Return a single scalar value (RETURNS INT, RETURNS TEXT, etc.).

    • Return a single row (RETURNS record).

    • Return a set of rows (RETURNS SETOF record or RETURNS TABLE).

    • Perform data modifications (INSERT, UPDATE, DELETE).

    • Contain complex procedural logic (loops, conditionals, variable assignments).

    • Be written in various languages (PL/pgSQL, PL/Python, PL/pgSQL, etc.).

  • PostgreSQL also introduced the CREATE PROCEDURE command in version 11. These are similar to functions but are specifically designed for transactional control (e.g., they can commit or rollback transactions within the procedure, which functions cannot). They do not return a value (they are RETURNS VOID implicitly).

So, the DeptAvgSalary in your Canvas is a PostgreSQL function that behaves very much like what you might consider a "stored procedure" in other database systems, as it performs a calculation and returns a table, rather than just a single scalar value. It is indeed stored persistently in your PostgreSQL database.
